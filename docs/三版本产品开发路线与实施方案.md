# QuantMind 三版本产品开发路线与实施方案

文档版本：v1.0  
创建日期：2026-04-03  
适用范围：`quantmind` 主仓库（后端/前端/Electron/部署）

---

## 1. 背景与目标

QuantMind 后续将并行发展三个产品形态：
1. **开源版（OSS）**：本地部署优先，精简多用户与外部云服务依赖。
2. **社区版（Community）**：当前线上主版本，维持完整社区与多用户能力。
3. **企业版（Enterprise）**：多策略并行运行 + 高级因子 + 企业级治理能力。

核心目标：
1. 用一套核心代码维护三条产品线，避免长期三套分叉。
2. 控制版本复杂度，保证升级路径清晰（OSS -> Community -> Enterprise）。
3. 保证多租户与安全边界在 Community/Enterprise 中严格生效。

---

## 2. 版本定义与能力边界

## 2.1 开源版（OSS）

定位：本地单机可运行、低门槛、可二次开发。  
典型用户：个人开发者、研究者、小团队 PoC。

建议保留：
1. 单用户登录（可选关闭鉴权）。
2. **核心全功能支持：智能策略、回测中心、模型训练、QuantBot、多模型管理、模型推理（无功能限制）。**
3. 本地文件存储与本地模型管理。
4. 基础监控（日志 + 简单健康检查）。

建议移除/替换/限制：
1. **实时行情限制**：开源版不提供原生实时行情接口（需用户自行对接）。
2. 社区模块（帖子、评论、关注、通知中心）。
3. 短信服务与手机绑定。
4. COS 依赖（替换为本地磁盘或 MinIO 可选适配）。
5. 多租户隔离逻辑（OSS 固定单租户）。

部署形态：
1. `docker compose` 一键启动（本机 PostgreSQL/Redis）。
2. 默认配置不依赖公网云服务。

## 2.2 社区版（Community）

定位：面向 **Pro 专业用户** 提供的全链路系统解决方案。
典型用户：有实盘交易需求、追求行情实效性与功能完整性的量化投资者。

能力范围：
1. **全套功能支持**：提供 **智能策略、回测中心、模型训练、QuantBot、多模型管理、模型推理**（全功能无限制）。
2. **最新行情数据**：由运营团队负责，**支持 T-0 实时/最新完整数据**，满足专业实盘研报需求。
3. **独立安装包**：提供 Windows/macOS 独立完整安装程序，确保本地计算与存储的高效与隐私。
4. **Pro 方案基础**：作为后续高级功能（Pro 订阅）的核心载体，提供全闭环交易链路。
5. 多用户鉴权、用户中心、基础权限、社区内容。

## 2.3 企业版（Enterprise）

定位：面向金融机构（券商、公募/私募）提供的企业级量化基础设施。
典型用户：对数据质量、执行效率、安全性及多用户协作有极致要求的机构团队。

能力范围：
1. **机构版优化 (Institutional Optimization)**：支持 **多租户隔离、RBAC 精细化权限控制、完整的审计日志**，确保机构级合规。
2. **高价值数据 (High-value Data)**：包含 **L2 特种深度因子、146+ 维机构版特征字典、逐笔成交委托流数据**。
3. **高并发执行引擎**：原生支持 **100+ 策略并发运行、分布式 K8s 模型训练与多模型并行推理**。
4. **私有化集群部署**：主打对 K8s、Docker Swarm 的原生支持，保障机构核心数据主权。
5. **SLA 与 专属支持**：提供 7x24 小时技术保障服务及专属架构师咨询。
6. 更强可观测性（SLA、告警、容量规划看板）。

## 3. 统一架构策略（避免三套代码分叉）

## 3.1 架构原则

1. **Core + Extensions**：核心能力与版本扩展解耦。
2. **Feature Flags + Edition Gates**：通过能力开关控制版本功能暴露。
3. **单 API 契约**：不同版本尽量复用同一接口契约，按 capability 降级而非删除接口。
4. **配置驱动**：版本差异由配置与能力声明驱动，不靠硬编码分支散落。

## 3.2 目录与模块建议

建议逐步形成如下结构（可渐进迁移）：
1. `backend/core/*`：回测、训练、推理、策略执行、基础数据服务。
2. `backend/extensions/community/*`：社区内容、关注、通知、运营后台扩展。
3. `backend/extensions/enterprise/*`：多策略调度、企业权限、高级因子、审计增强。
4. `electron/src/core/*`：通用 UI 与服务层。
5. `electron/src/extensions/community/*`、`electron/src/extensions/enterprise/*`：版本差异页面与组件。

## 3.3 能力声明机制

新增统一能力接口（建议）：
1. `GET /api/v1/system/capabilities`
2. 返回示例：
   - `edition`: `oss | community | enterprise`
   - `features`: `{ community: true/false, sms: true/false, cos: true/false, multi_strategy: true/false, advanced_factors: true/false }`

前端基于 capabilities 控制菜单和页面入口，后端二次校验，防止绕过前端。

## 5. 开发分期与里程碑

## 5.1 Phase A：版本框架与 OSS 最小可用（2-3 周）

目标：先把“版本机制”做对，再做功能拆分。

任务：
1. 建立 `edition + capabilities` 配置框架。
2. 前后端接入能力开关，屏蔽社区/短信/COS 页面与接口入口。
3. OSS 本地存储替代 COS（文件路径、下载、上传统一适配）。
4. 提供 `docker-compose.oss.yml` 与 `.env.oss.example`。

验收：
1. OSS 单机可完成：启动 -> 训练 -> 推理 -> 回测。
2. 无 COS/短信依赖情况下不报错。

## 5.2 Phase B：社区版稳定化与双轨发布（3-4 周）

目标：确保 Community 主线不受 OSS 拆分影响。

任务：
1. 社区版回归测试清单与自动化脚本补齐。
2. API 契约对齐与废弃字段治理（保持向后兼容）。
3. 完成 `oss/community` 双轨镜像发布流水线。

验收：
1. 社区核心路径回归通过（鉴权、社区、通知、训练、推理、回测）。
2. 发布流程可一键产出两版本镜像。

## 5.3 Phase C：企业版首批能力（4-6 周）

目标：上线企业版差异化核心价值。

任务：
1. 多策略运行编排（任务队列、并发控制、策略优先级、资源配额）。
2. 高级因子扩展（新增 10+ 因子，版本化管理与血缘记录）。
3. RBAC 与审计增强（角色权限矩阵、关键操作审计日志）。
4. 企业版运维能力（资源告警、SLA 报表、容量评估）。

验收：
1. 并发策略运行稳定，具备隔离与限流。
2. 高级因子可在训练/回测/推理链路统一生效。

## 6. 数据与因子治理方案（重点）

## 6.1 因子统一原则

1. 训练、回测、推理统一使用同一因子版本。
2. 因子字典版本化（`qm_feature_set_*`）必须保持单一 active。
3. 禁止前端硬编码特征字段。

## 6.2 三版本策略

1. OSS：仅保留基础 146 因子（或可裁剪子集）。
2. Community：维持 146 因子主集，按稳定优先。
3. Enterprise：在 146 基础上增加高级因子集（单独版本号与回退机制）。

## 6.3 变更流程

1. 新增/删除/下线因子先进入 `draft`。
2. 跑完整离线评估（样本外指标 + 稳定性 + 回归）。
3. 通过后切 `active`，并记录变更审计。

## 7. 发布与部署策略

## 7.1 镜像与标签规范

建议统一：
1. `quantmind-api:oss-vX.Y.Z`
2. `quantmind-api:community-vX.Y.Z`
3. `quantmind-api:enterprise-vX.Y.Z`

其他核心服务同理（`trade/engine/stream`）。

## 7.2 配置模板

1. `.env.oss.example`
2. `.env.community.example`
3. `.env.enterprise.example`

规则：
1. 通过 `APP_EDITION` 声明版本。
2. 启动时校验版本必需配置，缺失即 fail-fast。

## 7.3 部署入口

1. OSS：`docker-compose.oss.yml`
2. Community：`docker-compose.server.yml`（现有）
3. Enterprise：`docker-compose.enterprise.yml` 或 K8s Helm Chart

---

## 8. 测试策略与验收门禁

## 8.1 测试分层

1. 单元测试：核心算法、配置解析、权限校验。
2. 集成测试：API + DB + Redis + 任务队列。
3. E2E 测试：前后端联调关键路径。

## 8.2 版本验收清单

OSS 验收：
1. 零云依赖启动成功。
2. 本地训练/推理/回测全通。

Community 验收：
1. 多用户与社区链路稳定。
2. 多租户隔离检查通过（含 `tenant_id/user_id` 必传校验）。

Enterprise 验收：
1. 多策略并发压测通过。
2. 权限/审计/高级因子全链路通过。

---

## 9. 风险清单与应对

1. **风险：版本分叉导致维护成本激增**  
应对：坚持 Core + Extensions，禁止直接复制模块。

2. **风险：配置复杂度增加导致部署失败**  
应对：提供版本模板、启动自检、文档化默认值。

3. **风险：因子版本不一致导致结果不可复现**  
应对：强制特征版本入参、模型元数据落盘、运行审计。

4. **风险：企业并发场景下资源争抢**  
应对：配额、限流、队列优先级、任务超时与熔断。

---

## 10. 组织与协作建议

建议按小队并行：
1. 平台组：版本框架、能力开关、发布流水线。
2. 量化组：因子治理、训练推理链路、高级因子。
3. 应用组：前端能力裁剪、社区功能、企业页面。
4. 运维组：部署模板、观测、告警、容量管理。

每周例会固定输出：
1. 三版本燃尽图。
2. 风险与阻塞项。
3. 跨版本变更清单（避免互相破坏）。

## 12. 附录：实施原则

1. 任何跨版本改动必须先定义“默认行为 + 降级行为”。
2. 对外 API 优先兼容，内部实现可迭代重构。
3. 版本差异必须可观测（日志中带 `edition`）。
4. 文档与代码同版本迭代，避免“代码变更、文档滞后”。
