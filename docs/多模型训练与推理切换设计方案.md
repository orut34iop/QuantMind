# 多模型训练与推理切换设计方案（全链路已落地）

## 1. 目标与结论
- 目标：打通“普通用户训练 -> 模型注册与本地镜像 -> 用户默认模型 + 策略级覆盖 -> 推理/策略激活统一生效”闭环。
- 已落地决策：
  - 切换粒度：`用户默认模型 + 策略绑定覆盖`。
  - 推理优先级：`显式 model_id > 策略绑定 > 用户默认 > model_qlib > alpha158`。
  - 产物接入：`models/candidates/{run_id}` 同步到 `models/users/{tenant}/{user}/{model_id}` 本地镜像。
  - 兼容策略：保留 `model_qlib -> alpha158` 现网兜底。

---

## 2. 数据模型

### 2.1 用户模型表 `qm_user_models`
固定字段：
- `tenant_id`
- `user_id`
- `model_id`
- `source_run_id`
- `status`（`candidate -> syncing -> ready -> active/archived -> failed`）
- `storage_path`
- `model_file`
- `metadata_json`
- `metrics_json`
- `is_default`
- `created_at`
- `updated_at`
- `activated_at`

约束：
- 访问隔离：所有查询强制 `tenant_id + user_id`。
- 默认唯一：同一用户仅允许一个 `is_default=true`（通过部分唯一索引保障）。

### 2.2 策略绑定表 `qm_strategy_model_bindings`
固定字段：
- `tenant_id`
- `user_id`
- `strategy_id`
- `model_id`
- `updated_at`

约束：
- 唯一键：`(tenant_id, user_id, strategy_id)`。

## 4. 训练完成到模型注册

训练回调成功后执行注册事务：
1. 从 `run_id` 生成稳定 `model_id`（`mdl_*`）。
2. 写入 `qm_user_models`，状态先置 `candidate -> syncing`。
3. 同步 COS 产物：
   - 源：`models/candidates/{run_id}/`
   - 目标：`models/users/{tenant_id}/{user_id}/{model_id}/`
4. 校验：
   - 模型文件存在（`model.lgb/model.txt/model.bin`）
   - `metadata.json` 完整可解析
   - 特征维度有效
   - `target_horizon_days/target_mode` 可追溯
5. 成功置 `ready`；失败置 `failed` 并写明确错误。
6. 首个业务可用模型自动设为默认模型。

训练状态查询结果新增：
- `result.model_registration = { model_id, status, error, storage_path, model_file }`

## 5. 推理与策略执行路由

### 5.1 统一解析器
`resolve_effective_model(tenant_id, user_id, strategy_id?, model_id?)`

优先级：
1. 显式 `model_id`
2. 策略绑定
3. 用户默认
4. 系统 `model_qlib`
5. 兜底 `alpha158`

### 5.2 推理接口变更
- `POST /api/v1/inference/predict`
  - `model_id` 改为可选
  - 新增可选 `strategy_id`
  - 响应新增：
    - `effective_model_id`
    - `model_source`
    - `fallback_used/fallback_reason`

### 5.3 策略激活链路
- `POST /api/v1/strategies/{strategy_id}/activate` 后触发推理时，按 `strategy_id` 自动解析生效模型。
- 触发链路统一回传实际生效模型与兜底信息。

### 5.4 Pipeline 推理分支
- `pipeline` 支持在未传 `model_id` 时自动按用户/策略解析。
- 显式 `model_id` 仍优先覆盖。

## 6. 前端联动

### 6.1 模型训练页
- 训练结果区显示 `model_registration` 同步状态（`syncing/ready/failed`）。
- 新增“设为默认模型”动作，直接调用用户态接口。
- 继续坚持“后端结果单一真源”：缺指标不伪造默认值。

### 6.2 策略管理页
- 新增“策略模型绑定”选择器：
  - 可选择 ready/active 模型；
  - 支持保存绑定与解除绑定；
  - 未绑定时回落用户默认模型/系统兜底。

---

## 7. 迁移与上线

### 7.1 一次性迁移脚本
- `scripts/migrate_user_model_registry.py`
  - 建表与索引
  - 基于存量训练记录回填 `model_qlib` 系统记录（只读标记在 `metadata_json`）

### 7.2 启动时幂等保障
- `quantmind-api` 与 `quantmind-engine` 启动均会幂等执行模型注册表结构校验。

---

## 8. 验收标准
- 普通用户可训练并查询自己的训练任务。
- 训练完成后自动生成用户模型记录并可在本地目录加载。
- 跨用户访问模型/绑定被拒绝（`tenant_id + user_id` 强隔离）。
- `predict` 在四种场景下命中正确模型：
  - 显式 `model_id`
  - 策略绑定
  - 用户默认
  - 系统兜底
- 未配置任何用户模型时，保持 `model_qlib -> alpha158` 可用。
