# 高维特征存储与统一访问方案 (146-Dimension Feature Store)

## 1. 方案背景
随着模型复杂度的提升，系统需支撑 150 个维度的特征数据。为了确保**训练 (Training)**、**回测 (Backtest)** 与 **推理 (Inference)** 使用完全一致的数据源，并解决高维数据带来的性能瓶颈，本方案提出了一套基于“特征平台 (Feature Store)”理念的存储与访问架构。

核心目标：**一份真值 (Single Source of Truth)，多态输出 (Multi-Modal Output)**。

---

## 2. 存储矩阵 (Storage Matrix)

针对不同场景的性能需求，采用分层存储策略：

| 场景 | 存储介质 | 数据格式 | 核心优势 |
| :--- | :--- | :--- | :--- |
| **离线训练** | Qlib Binary | 按列存储的 `.bin` 文件 | 极速全量扫描，Qlib 原生支持，内存占用低。 |
| **大规模回测**| Parquet / ClickHouse | 列式存储 | 支持时空跨度查询，方便进行特征分布分析。 |
| **实时推理** | Redis | Hash (Key-Value) | <1ms 延迟，仅存每个 Symbol 的最新一行特征。 |
| **原始凭据** | PostgreSQL | JSONB / 关系型表 | 事务安全，作为所有派生数据的“真值源”。 |

---

## 3. 数据链路设计 (Data Pipeline)

### 3.1 离线链路：训练与回测
1. **真值落库**：特征计算服务器将 146 维特征写入 PostgreSQL 的 `market_data_daily`。
2. **格式转换 (ETL)**：每日收盘后，自动触发同步脚本，将 PG 数据转换为 Qlib 二进制格式。
   - 路径：`db/qlib_data/cn/features/`
   - 文件名：`feat_001.bin` ... `feat_150.bin`
3. **一致性保证**：回测引擎通过 Qlib `DataHandler` 直接读取这些二进制文件，确保回测所见即训练所得。

### 3.2 在线链路：实时推理
1. **实时推送到 Redis**：
   - 特征服务器在产出最新特征后，立即更新 Redis。
   - Key 设计：`qm:feature:live:{symbol}`
   - 数据结构：`Hash { "f1": 0.5, "f2": -1.2, ... "f150": 0.8 }`
2. **推理访问**：`InferenceService` 不再实时计算特征，而是直接从 Redis 读取这 150 个预计算值。

## 4. 统一访问接口 (FeatureProvider)

在后端引入 `FeatureProvider` 抽象层，屏蔽底层存储差异：

```python
class FeatureProvider:
    """统一特征访问入口"""
    
    def get_latest_row(self, symbol: str, feature_list: List[str]) -> np.ndarray:
        """[推理使用] 从 Redis 获取最新一行特征"""
        raw = redis.hmget(f"qm:feature:live:{symbol}", feature_list)
        return np.array(raw, dtype=np.float32)

    def get_time_series(self, symbol: str, start_date: str, end_date: str) -> pd.DataFrame:
        """[回测使用] 从 Qlib/Parquet 获取时序特征"""
        return qlib.get_data(symbol, start_date, end_date)
```

在模型的 `metadata.json` 中明确定义特征索引，实现“特征按需加载”：

```json
{

  }

```

## 6. 实施路线图

1. **第一阶段：特征表扩展**
   - 优化 PostgreSQL `market_data_daily` 表索引，支持 146 维 JSONB 高速写入。
   - 建立 `db/qlib_data/` 的特征列目录规范。

2. **第二阶段：自动同步工具**
   - 开发 `pg_to_qlib_bin.py`：实现 PG 到 Qlib 二进制的高效转换。
   - 开发 `pg_to_redis_live.py`：实现 Redis 实时特征缓存刷新。

3. **第三阶段：推理引擎接入**
   - 修改 `DataAdapter`，支持从 `FeatureProvider` 获取特征，而非本地重算。
   - 在 `quantmind-engine` 中集成 Redis 特征读取逻辑。

---

## 7. 风险点与应对
- **Redis 内存压力**：146 维 * 5000 只股票 ≈ 75 万个浮点数。
  - *应对*：使用紧凑的 Hash 存储，或仅保留活跃股票（如全 A 股）的特征。
- **数据版本冲突**：训练时用的特征定义发生变更。
  - *应对*：在特征库中引入 `version` 标签，模型绑定特定版本的特征库。
